# CSS / UI 개선 설계서

## 목적
이 문서는 현재 `apps/web`의 UI/CSS 구조를 코드 수정 없이 분석하고, 다른 AI가 작업 중인 영역을 건드리지 않으면서도 이후 안전하게 개선할 수 있도록 방향을 정리한 설계 문서다.

---

## 현재 구조 요약

### 앱 성격
현재 웹앱은 다음 두 성격이 섞여 있다.

1. 서비스 소개용 랜딩 페이지
2. 로그인 후 사용하는 운영형 백오피스(관리자/실무자 시스템)

이 방향 자체는 가능하지만, 현재는 두 성격이 같은 디자인 언어 안에서 충분히 분리되지 않아 UI 목적성이 흐려질 수 있다.

### 주요 파일
- `apps/web/app/globals.css`
- `apps/web/app/layout.tsx`
- `apps/web/components/shell.tsx`
- `apps/web/app/dashboard/page.tsx`
- `apps/web/app/login/page.tsx`

### 현재 특징
- 전역 CSS 중심 구조
- 부드러운 베이지/민트/코랄 톤
- 유리 질감, blur, 큰 그림자 사용
- hover 기반 확장 사이드바
- 다수의 운영 메뉴를 포함한 백오피스 구조

---

## 핵심 문제 진단

### 1. 전역 CSS 집중도 과다
`globals.css`에 레이아웃, 컴포넌트, 페이지 스타일이 광범위하게 포함되어 있다.

#### 리스크
- 특정 수정이 다른 화면에 예상치 않게 영향 줄 수 있음
- 디버깅 난이도 상승
- 여러 AI/개발자 협업 시 충돌 증가
- 파일이 커질수록 유지보수 비용 급증

#### 판단
초기 프로토타입으로는 괜찮지만, 제품화 단계에서는 구조 분리가 필요하다.

---

### 2. 랜딩 감성과 업무 시스템 감성의 충돌
현재 비주얼은 따뜻하고 감성적이며 보기 좋지만, 업무형 시스템에서 더 중요한 것은 즉시성, 안정감, 정보 우선순위, 빠른 인지다.

#### 문제 양상
- 장식적 요소 비중이 높음
- 업무 화면까지 감성형 표현이 유지될 가능성 높음
- 실제 실무 데이터가 많아질수록 산만해질 수 있음

#### 판단
랜딩과 백오피스는 같은 브랜드를 공유하더라도 UI 밀도와 시각적 톤은 분리해야 한다.

---

### 3. 사이드바 UX 불안정
`Shell` 컴포넌트는 hover 또는 pin 상태에 따라 레일이 접히고 펼쳐지는 구조다.

#### 문제 양상
- hover 기반 사이드바는 실무 시스템에서 피로를 줄 수 있음
- 접힘 상태에서 메뉴가 첫 글자만 보여 의미 전달이 약함
- 메뉴 수가 많아질수록 탐색성이 떨어짐
- 장시간 업무 시 정보 구조보다 상태 전환에 더 신경 쓰게 될 수 있음

#### 판단
운영 시스템은 예측 가능하고 안정적인 내비게이션이 우선이다.

---

### 4. 정보 구조 그룹핑 부족
현재 메뉴 개수는 많은데, 사용자가 인지하기 좋은 상위 그룹 구조가 UI에서 명확하게 드러나지 않는다.

#### 문제 양상
- 기능이 나열형으로 느껴짐
- 핵심 업무와 보조 기능이 동급처럼 보임
- 신규 사용자 진입 장벽 상승

#### 추천 그룹 예시
- 운영관리
- 사례/아동관리
- 인사/근태
- 문서/결재
- 커뮤니케이션
- 시스템/보안

---

### 5. 대시보드가 실무형보다 데모형에 가까움
대시보드는 카드, 액션 버튼, 추이 요약으로 구성되어 있으며 보기에는 좋다. 다만 실제 운영 관점에서는 우선순위와 처리 중심성이 더 필요하다.

#### 부족한 요소
- 오늘 처리해야 할 일
- 긴급도/위험도 구분
- 승인 대기/미처리 항목 강조
- 최근 활동 및 알림 흐름
- 업무로 곧바로 연결되는 액션

#### 판단
현 상태는 소개형 관리자 화면에 가깝고, 운영형 대시보드로 발전시키려면 상태 중심 설계가 필요하다.

---

### 6. 타이포/공간 밀도 문제 가능성
헤드라인이 큰 편이고 카드 여백도 넉넉하다. 랜딩에서는 장점이지만 내부 업무 화면에서는 정보 밀도를 떨어뜨릴 수 있다.

#### 영향
- 스크롤 증가
- 한 화면에 보이는 정보량 감소
- 표/폼/리스트 중심 화면에서 비효율 발생

---

### 7. 시각 효과 과다 사용 우려
blur, 투명 배경, 큰 shadow, gradient가 비교적 적극적으로 사용된다.

#### 리스크
- 저사양 환경 성능 저하 가능성
- 시각 피로도 증가
- 표/폼 중심 화면에서 집중력 저하

---

### 8. 로그인 화면의 데모 인상
샘플 계정 정보가 UI에 직접 노출된다.

#### 영향
- MVP/시연 목적이라는 인상이 강해짐
- 실서비스 톤 약화
- 실제 배포 시 보안 인식에 악영향

## 우선순위 정리

### P0 — 가장 먼저 방향을 정해야 하는 것
1. 랜딩과 백오피스의 디자인 역할 분리
2. 사이드바 UX 정책 결정
3. 메뉴 그룹 구조 확정

### P1 — 구조 정리
1. 글로벌 CSS 역할 축소
2. 공통 컴포넌트 스타일 규칙 정의
3. 페이지/섹션별 스타일 분리 계획 수립

### P2 — 운영형 UI 강화
1. 대시보드 상태 중심 재구성
2. 액션 우선순위 재설계
3. 폼/테이블/배지 규칙 추가

### P3 — 마감 품질
1. 모바일 대응 점검
2. 명도 대비 및 접근성 점검
3. 미세 인터랙션 정리

## 수정 없이 바로 적용 가능한 운영 원칙
다른 AI가 코드를 만지고 있는 상황을 고려해, 당장은 아래 원칙만 공유해도 협업 충돌을 줄일 수 있다.

### 원칙 1. 기존 공용 클래스 이름을 함부로 재정의하지 않는다
- 특히 `globals.css`에 있는 공통 클래스는 영향 범위가 넓다.
- 임시 수정이 필요하면 새 네이밍을 우선 고려한다.

### 원칙 2. 업무 화면과 랜딩 화면을 같은 미적 기준으로 보지 않는다
- 랜딩: 브랜드/첫인상
- 내부앱: 효율/가독성/처리속도

### 원칙 3. 메뉴는 늘리기보다 묶는다
- 메뉴를 추가하기 전에 상위 카테고리 구조부터 본다.

### 원칙 4. 대시보드는 꾸미기보다 우선순위 표시에 집중한다
- 숫자 카드보다 처리할 일, 미처리, 알림, 위험 신호가 먼저 보여야 한다.

### 원칙 5. 스타일 수정 전에 영향 범위를 문서화한다
- 어떤 클래스가 어디에 쓰이는지 먼저 기록하고 건드린다.

## 권장 리팩터링 방향 (후속 작업용)

### 단계 1. 스타일 계층 분리
권장 방향:
- `globals.css`: reset, tokens, body/html, truly global primitives only
- shell/navigation 관련 스타일 분리
- auth 관련 스타일 분리
- dashboard 관련 스타일 분리
- 문서/테이블/폼 공통 컴포넌트 스타일 분리

예시 구조:
- `styles/tokens.css`
- `styles/base.css`
- `styles/shell.css`
- `styles/auth.css`
- `styles/dashboard.css`
- `styles/components/*.css`

또는 Next.js 기준으로 CSS module/component 단위 전환도 가능하다.

### 단계 2. 디자인 토큰 고정
우선 명시해야 할 것:
- primary / secondary / danger / warning / success 색상
- 배경 레벨 1/2/3
- border radius 단계
- shadow 단계
- spacing scale
- title/body/caption 타입 스케일

### 단계 3. 내비게이션 체계 재설계
권장 방향:
- 고정형 사이드바 기본
- 필요 시 collapse는 명시적 버튼으로만
- 메뉴 그룹 제목 제공
- 아이콘+텍스트 병행
- 하단에 시스템/설정/로그아웃 분리

### 단계 4. 운영형 대시보드로 전환
대시보드 구성 예시:
1. 오늘 꼭 처리할 일
2. 긴급 알림/마감 임박
3. 주요 운영 지표
4. 최근 활동 로그
5. 빠른 작업 버튼

### 단계 5. 컴포넌트 표준화
우선 표준화 대상:
- 버튼
- 카드
- 배지
- 입력 필드
- 표/리스트 행
- 패널 헤더
- empty state
- 상태 알림

## 다른 AI와 병행 작업 시 안전한 분업안

### 다른 AI가 맡아도 되는 영역
- 실제 기능 구현
- 라우트 추가
- 데이터 연결
- 인증/권한 로직 확장

- UX 문제점 정리

### 충돌 회피 방법
- 코드 수정 전 대상 파일 미리 선언
- 공용 CSS 직접 수정은 마지막 단계로 미룸
- 새 파일/새 컴포넌트 우선 전략 사용
- 한 번에 큰 CSS 변경보다 단계별 변경

## 결론
현재 UI는 방향성이 나쁘지 않다. 다만 제품이 성장하면 가장 먼저 병목이 될 가능성이 높은 것은 다음 세 가지다.

1. 과도한 전역 CSS 의존
2. 랜딩 감성과 업무 UI의 혼합
3. 탐색 구조와 우선순위 표현의 미완성

즉, 지금 필요한 것은 무작정 예쁘게 고치는 것이 아니라, 운영형 제품으로서의 정보 구조와 스타일 체계를 먼저 굳히는 일이다.
